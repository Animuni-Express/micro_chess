Third pass (post-merge live bug report — "Game access denied"):
- Root cause: the "COPY LINK" button on a waiting room shares a URL keyed by game id (?game=<uuid>). A friend opening that link is a fresh guest who isn't a recognized player yet, so get_game rejects them with "Game access denied" — there was never a way to actually JOIN via that link, only the separate room-code text field worked.
- Fix: added a join_multiplayer_game_by_id RPC (new migration 20260723150000_join_by_game_id.sql, mirrors join_multiplayer_room but keyed by id) and a join_by_id Edge Function action. Both clients (desktop OnlineGameApp.jsx, Kindle KindleOnlineGameApp.jsx) now fall back from get_game to join_by_id when access is denied AND the player has never been seated in this game before (tracked via a ref so a later, unrelated denial is never misread as an invite).
- Added Playwright coverage for the exact bug: a second browser context opens the shared link directly (never touching the room-code field) and must end up seated with the opposite colour, no "access denied" message. 44/44 checks green (16 functional + 23 polyfill + 5 Kindle-render), stable across repeated runs.

Fourth pass (live bug report — Kindle board size, Kindle online render, matchmaking, latency):
- Kindle board smaller than intended: an earlier Kindle iteration explicitly sized cells at 72px for "larger tap target for Kindle touch"; the Preact rewrite quietly shrank `.board td` to 56px. Bumped back up to 64px (with `.board td img` 44→52px, `.piece-fallback` 40→46px, and `.container` max-width 500→560px so the larger board still fits) across all three Kindle modes (OTB, Stockfish, Online), which all share this CSS.
- Online multiplayer never rendered on Kindle: `KindleOnlineGameApp.jsx` used `URL`/`URLSearchParams` to read and rewrite the `?game=` query param. Old Kindle WebKit has neither, and unlike every other runtime gap in this codebase (Promise, fetch, Set, padStart, etc.) it was never polyfilled — so the very first render threw and the page stayed blank. Replaced with plain string parsing/rebuilding (no polyfill needed, matches the ES5-safe style already used elsewhere in this file).
- "Find a Match" landing on a pre-existing room/game instead of a fresh match (confirmed on all devices, so a backend bug, not Kindle-specific): `matchmaking_status` picked the player's most-recently-updated game in ('waiting','active') — which could be a stale private room they'd created earlier, or an old active game, not necessarily an actual new match. Fixed by tracking the match on the matchmaking_queue row itself: `queue_multiplayer_player` now sets `matched_game_id` on the opponent's queue row (new migration 20260723160000_fix_matchmaking_status.sql) instead of deleting it, and `matchmaking_status` only ever reports a match by reading that row — never any other game the player happens to be seated in.
- Latency higher than the already-documented Tokyo-distance baseline: every mutating action awaited `broadcastGame` before responding, which subscribes a brand-new Realtime channel per call and can take up to 800ms to confirm — the acting player was blocked on a broadcast meant for their opponent. Made it fire-and-forget via `EdgeRuntime.waitUntil` so the response returns immediately; broadcastGame already swallows its own errors, so nothing is lost by not awaiting it.
